프로젝트를 진행하다 보면 "커밋은 했는데 나중에 왜 했는지 모르겠다" "어느 시점에서 오류가 발생했는지 모르겠다" 등 커밋 메시지만으로 코드의 변경 내용을 파악하기 어려운 상황이 자주 발생하였습니다.
문제가 발생하는 원인은 **불규칙적인 커밋 메시지** 작성으로 인해 발생하였고 **좋은 커밋 메시지 작성 가이드라인**의 필요성을 느꼈습니다.

## 불규칙적인 커밋 메시지 문제점

당장은 별 문제가 없어 보여도, 시간이 지나면 아래 비용이 눈덩이처럼 커집니다.

- **변경 의도 파악 비용 증가**
  - “왜 바꿨지?”를 코드 diff만 보고 추측해야 해서 리뷰/유지보수 시간이 늘어납니다.
- **히스토리 탐색이 어려움**
  - `git log`, `git blame`로 문제를 추적할 때 단서가 부족해서 원인 파악이 늦어집니다.
- **릴리즈 노트/변경 로그 자동화 불가**
  - 커밋이 일관되지 않으면 Conventional Commits 기반 자동 changelog 생성이 사실상 불가능합니다.
- **코드 리뷰 품질 저하**
  - 메시지가 내용/범위를 설명하지 않으면 리뷰어는 “이 커밋이 뭘 해결하려는지”부터 다시 읽어야 합니다.
- **롤백/핫픽스 대응 속도 저하**
  - 장애 상황에서 관련 커밋을 빠르게 찾고 되돌려야 하는데, 메시지가 모호하면 대응이 늦어집니다.

## 좋은 commit message 작성법

### commit message 구조

**Type 기반** commit message 구조

```text
<type>(<scope>): <subject>

<body>

<footer>
```

- **type**: 변경 성격 (feat, fix 등)
- **scope(선택)**: 영향을 받는 모듈/도메인/패키지 등 (auth, api, ui 등)
- **subject**: 한 줄 요약 (명확하고 간결하게)
- **body(선택)**: 왜/무엇을/어떻게 바꿨는지 추가 설명
- **footer(선택)**: 이슈 링크, Breaking change, 참고사항 등

```text
feat(auth): 소셜 로그인 콜백 처리 추가

- 카카오/구글 콜백 엔드포인트 추가
- 신규 유저 가입 시 기본 권한 부여

Refs: #123
```

### 기본 규칙

#### Subject 규칙 (한 줄 요약)

- **명령형**으로 쓰기
  - ✅ `feat: 로그인 버튼 추가`
  - ✅ `design: 로그인 버튼 색상 수정`
  - ✅ `fix: 토큰 만료 시 재발급 처리`
- **무엇을 했는지 한 줄로 이해 가능**하게
  - ❌ `update` / `misc` / `temp`
- **50자 내외** 권장
- 끝에 마침표(.) 사용 금지
- 어떻게 보다는 무엇과 왜
- 누가 봐도 그 커밋이 어떤 변화인지 드러나야 함

#### Body 규칙 (필요할 때만)

아래 중 하나라도 해당되면 body를 쓰는 걸 추천합니다.

- 변경 이유가 코드만 봐선 잘 안 보인다.
- 버그 원인과 해결 방식이 중요하다.
- 롤백/마이그레이션/주의사항이 필요하다.

Body에는 보통 아래 포맷이 잘 먹힙니다.

- **왜(문제/배경)**
- **무엇(변경 내용)**
- **영향/주의점(사이드 이펙트, 마이그레이션)**

예시:

```text
fix(api): 페이징 offset 계산 오류 수정

기존 offset 계산이 1-based로 처리되어 첫 페이지에서 중복 데이터가 반환됨.
0-based 계산으로 통일하고, 관련 테스트 케이스 추가.

Refs: #201
```

#### Scope 규칙 (선택이지만 권장)

- scope는 “어디를 건드렸는지”를 빠르게 알려줍니다.
- 너무 세분화하면 복잡하여 아래 목록을 기준으로 나누었습니다.

예시 scope: `auth`, `ui`, `api`, `db`, `infra`, `docs`

#### Type 규칙

아래 Type 목록을 기준으로 사용합니다.

- **feat**: 새로운 기능 추가
  - `feat(user): 프로필 이미지 업로드 추가`
- **fix**: 버그 수정
  - `fix(payment): 결제 실패 시 재시도 로직 수정`
- **docs**: 문서 변경 (README, 위키 등)
  - `docs: 로컬 실행 방법 업데이트`
- **style**: 코드 스타일 변경 (포맷팅, 세미콜론 등) — 로직 변화 없음
  - `style: lint 규칙에 맞게 포맷 정리`
- **design**: UI 디자인 변경 (CSS/스타일 중심)
  - `design(ui): 버튼 색상 및 간격 조정`
- **test**: 테스트 코드 추가/수정
  - `test(api): 회원가입 API 테스트 추가`
- **refactor**: 리팩토링 (기능 변화 없이 구조 개선)
  - `refactor(auth): 토큰 검증 로직 모듈 분리`
- **build**: 빌드 시스템/의존성 관련 변경
  - `build: vite 플러그인 설정 추가`
- **ci**: CI 설정 변경 (GitHub Actions 등)
  - `ci: 테스트 워크플로 캐시 적용`
- **chore**: 기타 잡무 (설정, 작업, 코드 외 변경)
  - `chore: eslint 규칙 업데이트`
- **rename**: 파일/변수/모듈 이름 변경(의미 변화 없음이 핵심)
  - `rename: UserDto -> UserResponse로 변경`
- **remove**: 기능/파일 제거
  - `remove: 사용하지 않는 레거시 엔드포인트 삭제`

> 참고: `style` vs `design`
>
> - `style`은 **코드 포맷/컨벤션**(prettier, lint)
> - `design`은 **UI/UX 시각적 변경**(CSS, 간격, 색상)  
>   으로 구분하면 혼선이 줄어듭니다.

## 적용 후기

가이드라인을 적용한 뒤 가장 먼저 체감한 변화는 코드 리뷰 속도였습니다. 커밋 메시지만 읽어도 “이번 변경이 무엇을 위한 것인지”를 먼저 파악할 수 있어, diff를 볼 때 불필요한 해석 시간이 줄었습니다. 그 결과 리뷰 코멘트도 더 핵심에 집중하게 되었고, 전체적인 리뷰 대화의 길이와 시간도 자연스럽게 단축되었습니다.

다만 아무리 좋은 커밋 메시지 가이드라인을 만들어도 **일관성이 없다면 가이드라인의 의미가 없어집니다.** 결국 중요한 건 규칙 자체가 아니라, 팀원 모두가 **일관성 있게** 작성해 신뢰할 수 있는 히스토리를 쌓아가는 것이라고 생각합니다.
